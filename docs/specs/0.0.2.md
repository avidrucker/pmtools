# pmtools 0.0.2 — lifecycle completeness

**Status:** proposed (spec for the next release).
**Baseline:** 0.0.1 = the current `main` (`CONTRACT.md` as it stands at the 0.0.2
proposal). pmtools carried no version number before this; 0.0.1 is the retroactive
name for today's behavior.
**Living contract:** `CONTRACT.md` remains the language-neutral behavioral source of
truth. This file records the *delta* 0.0.2 introduces; once the items land, their
detail lives in `CONTRACT.md` and this file is the historical record of the release.
**Parity rule (unchanged):** every behavior change edits `CONTRACT.md` + a fixture, then
makes *both* ports (`js/`, `py/`) green. Ports never diverge silently.

## Theme

Complete the ticket lifecycle at **both ends**. 0.0.1 owns the middle — claim, preflight,
close, release, sweep — but assumes a ticket is born by hand and always closes on a code
commit. 0.0.2 adds the two missing ends:

- **`pmtools file`** — a gated *front door* for creating a ticket (from the #109 spike:
  "should pmtools own issue creation to gate on requirements?").
- **no-code close** — a *back door* for closing a comment-only ticket (research spike,
  decision ruling, triage) whose deliverable is a comment, not a commit.

Around those two flagships it ships seven cheap, high-confidence fixes that close
spec-vs-code gaps and finish in-flight consistency work.

## 1. `pmtools file` — gated issue creation (#109)

### Command

```
pmtools file --title T --area AREA [--role R] [--body-file F] [--label L ...] [--host github]
```

Creates a GitHub issue **through the provider adapter**, applying pre-create requirement
gates, then prints the verified issue number. Fits the existing lifecycle family
(claim / preflight / close / release / sweep): one token added to the `bin/pmtools`
dispatcher allowlist + a help stanza; a pure seam `file_core.{js,py}` graded against
`fixtures/file/*`; the impure `file.{js,py}` does only the provider I/O + wiring.

### Provider

Adds a **second provider write method**, `createIssue(...)`, next to the existing
`editIssueBody`. GitHub maps it to `gh issue create`; GitLab stays a stub. This keeps
the command host-agnostic and keeps all `gh` knowledge inside the adapter.

### Why it is worth building (the load-bearing reason)

A single `pmtools file` invocation **serializes** issue creation and **reads the number
back** from the provider's response. That structurally enforces the already-ratified
pycats#541 rule ("mint IDs one at a time, never concurrently; verify a number before
stating it") for the creation path — something a server-side label check cannot do,
because by the time a server-side hook runs, the number is already minted. The area-label
gate is a bonus; the race fix is the reason.

### Requirement gates (tiered, config-driven)

Gates read from a new `file` block in `.claude/orchestrate.json` (see §Config). They are
tiered by how crisply a machine can judge them; only label-based checks gate **hard**.

- **Tier 1 — hard block (default on in 0.0.2):**
  - *Serialized create + number read-back.* Inherent to the command; not a toggle.
  - *Exactly one `area:*` label.* Reuses the existing lane seam
    (`needsAreaLabel` / `shouldBlockUncategorized` in `claim_core`), **extended** to also
    reject two-or-more `area:*` (today's seam only rejects none / `area:uncategorized`).
- **Tier 2 — config-gated block (default OFF in 0.0.2):**
  - Title hygiene: em-dash-not-colon, banned-word list.
  - `severity:*` only on defects (a feature must carry no `severity:*`).
- **Tier 3 — warn only (never blocks):**
  - Role tag present (roles live in the body, not a label — fuzzy).
  - Body shape (`have / should / repro`) — a heuristic that would false-reject legitimate
    research/docs tickets if hard-gated.

`area:uncategorized` at file time is treated the way `claim` treats it: **allowed but
noticed**, so a genuinely-unsure filer is never hard-blocked.

### 0.0.2 scope decision

Ship **Tier-1 enforced + the full config surface for Tier-2/3 defaulting off/warn**. The
plumbing and the config keys all land in 0.0.2; the strict Tier-2/3 gates are opt-in per
project. The valid-area set lives in the `file` block's `areas:` key for now; when
#69 (`pmtools labels suggest`, deferred to the roadmap) ships, that becomes the single
taxonomy source and `file` reads from it instead of a duplicated list.

### Routing discipline

A front door only helps if agents use it. When `pmtools file` lands, add one line to the
consumer's RULES: *file via `pmtools file`, not raw `gh issue create`* — mirroring how
pycats#541 already constrains creation. Without that routing line the gate sits unused.
This is a consumer-repo action, not a pmtools code change; noted here so it isn't lost.

### Relationship to the server-side gate

`pmtools file` is the **ergonomic front door** (opt-in, catches the agent path, uniquely
fixes the minting race). The complementary **server-side catch-all** — a CI check that
flags an area-less issue no matter how it was filed — is pycats#570's decision to build,
**not** this spec's. The two are layered and complementary; 0.0.2 ships only the front
door.

### Exit codes

`0` on a successful create; a **usage error** (missing `--title`/`--area`, unknown flag)
→ `2`; an **operational** failure (a hard gate rejected the request, or the provider
create failed) → `1`. Matches the cross-command convention in `CONTRACT.md`.

## 2. no-code close — `close --no-code`

### Command

```
pmtools close <N> --no-code [--comment-file F | --comment "…"] [--no-comment]
```

Closes a ticket whose deliverable is a **comment**, not a commit — a research spike, a
decision ruling, a triage summary. The mirror image of `pmtools file`: `file` gates the
birth, no-code close handles the comment-only death.

### What it changes vs. a normal close

`close` normally demands a `Closes #N` commit on `origin/main`, lands it, then tears down
the worktree, guarding on markers/keywords/the verify gate — all of which assume changed
code. `--no-code`:

- **Skips** the commit scan (no `Closes #N` required), the marker-deleted guard, the
  keyword spot-check, and the pre-close verify gate.
- **Posts the closing comment** (the findings/ruling) via the provider, then closes the
  issue.
- **Still** logs velocity (roles RESEARCH / SPIKE / ARC / PM / REVIEW) — a no-code close is
  still a logged session — and **still** sweeps the claim ref.
- **Works with or without a worktree.** If one was claimed to do the research, it is torn
  down (reusing close's teardown). If the ticket was commented on directly with no
  worktree, it just closes + sweeps + logs — reusing `release`'s "no worktree → still free
  the ref" path.

### Design decisions

1. **A comment is required unless `--no-comment` is explicit.** The comment *is* the
   deliverable for these tickets; requiring it (with an explicit override) mirrors close's
   guard philosophy.
2. **The velocity guard still applies.** It already no-ops when `storage.velocity` is
   disabled, so no special-casing — the "every close logs a session" invariant holds.

### Reuse

Reuses close's pure seams and teardown machinery; the new surface is a `no_code` branch in
the flow plus a comment-post step. `CONTRACT.md` §close gains a "no-code mode" subsection;
`fixtures/close/*` gains cases for the guard-skips and the require-comment rule. Both ports
stay faithful twins.

### Exit codes

`0` on a successful no-code close; a **usage error** (bad flag combination, e.g. a comment
source *and* `--no-comment`) → `2`; an **operational** failure (missing comment without
`--no-comment`, provider close failed, velocity guard failed) → `1`.

## 3. `ice --set-tier` — priority override label (#108)

Adds an opt-in escalation flag to the shipped `ice` command:

```
pmtools ice --set-tier critical|elevated <issue> [--dry-run]
```

Applies a `priority:critical` or `priority:elevated` override label that sits **above** the
normal ICE queue, with an audit comment recording who set it and when. `critical` is
**human-only** (SLA/legal/all-agents-blocked); `elevated` is human-or-PM-agent. No change
to the automatic scoring path. New `fixtures/ice/*` cases; both twins. Parent epic #100
(`pmtools ice`) is closed; this is the deferred `--set-tier` follow-up from #103.

## 4. Pre-close verify-gate per-command timeout (#107)

Follow-up to the config-driven pre-close verify gate (#106, landed). Adds an optional
per-command wall-clock limit so a hung verify command can't wedge `close` indefinitely:

```jsonc
"close": {
  "verify": {
    "commands": ["ruff check .", "pytest -q"],
    "timeoutSec": 300        // NEW: per-command limit; unset/0 => no limit (today's behavior)
  }
}
```

A command exceeding the limit is killed (process-group kill, since verify commands run
under a shell) and treated as a **failure** — the same abort-and-preserve-worktree path as
a non-zero exit. Both twins.

## 5. close/release teardown `--force` (#82) — already resolved, no action

Investigated: **no divergence exists** on current `main`. Contract and code already agree at
both teardown sites, and the close-vs-release difference is intentional, not a bug.

- **close** — `CONTRACT.md` §close teardown step and the code (`js/close.js`, `py/close.py`)
  both call `git worktree remove` **without** `--force`. Correct: close tears down only after
  the clean-tree guard has passed and the commit has landed, so the worktree is clean and a
  plain `remove` succeeds. (The close teardown text was reworded to drop `--force` during the
  #104 close-from-main rework, which resolved the half of #82 that ever applied.)
- **release** — `CONTRACT.md` §release teardown and the code (`js/release.js`,
  `py/release.py`) both call `git worktree remove --force`. Correct and required: `release`
  deliberately discards a possibly-dirty / unpushed worktree (gated by its own `--force`
  flag), which a plain `remove` would refuse.

So #82's premise — both ports omit `--force` against a contract that mandates it — does not
hold today. The earlier tentative "add `--force` (option b)" plan is dropped: adding it to
`close` would be speculative hardening against a guard that is not relaxed, not a fix. #82 is
closed as non-reproducing.

## 6. preflight claim/close config-coherence warn (#63)

`preflight` emits a non-blocking `note:` when the consumer's config pairs a pmtools claim
with a non-pmtools close — i.e. `enrichment.claimCommand` resolves to `pmtools claim`
(which mints `br-`/`wt-` self-describing branch names) but `enrichment.closeCommand` does
**not** resolve to a pmtools close that can parse those names. This surfaces the #58 close
rejection at *start* time instead of after the work is done.

Requires **adding `enrichment.claimCommand` and `enrichment.closeCommand` config keys** —
today the `enrichment` block only carries `statusCommand` and `clusterFile`. Warn, never
block (a consumer may legitimately keep a non-pmtools close). Both twins.

## 7. Residual `warn:`/`warning:` → `note:` channel (#62)

Finishes the #44 diagnostic-channel unification on the warn side. Renames the 8 residual
sites — `claim`'s `[claim] warn:` and `close`/`release`'s `[<cmd>] warning:` — to the
uniform `[<cmd>] note:` warn channel, so one `grep -F '] note:'` finds every warning.
Pure string edits, no exit-code or behavior change; update the CONTRACT output-conventions
note (the "residual" caveat is removed). Both twins.

## 8. Velocity role glosses in CONTRACT (#95)

Adds a one-line gloss next to each of the 11 velocity `VALID_ROLES`
(`DEV, TEST, WRITER, RESEARCH, SPIKE, ARC, PM, COMBO, DATA, CHORE, REVIEW`) so an agent
picks the right role instead of guessing (the `TST`≠`TEST` trap). The vocabulary stays
closed and hard-reject; only its *meaning* is written down. Docs only — no code, no
fixture.

## 9. `close --worktree-dir` marked deprecated (#73)

After #51, `close` self-discovers the worktree via `git worktree list` and no longer uses
`--worktree-dir`, but the flag is still parsed and still advertised as a real option.
0.0.2 documents it as **deprecated / back-compat-only** in the close help text, `README.md`,
and `CONTRACT.md`: accepted without error (so older scripts don't break) but explicitly not
to be used going forward. Docs only.
